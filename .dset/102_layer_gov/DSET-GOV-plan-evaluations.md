# Methodology GOV qualitative eval plan

Applicable qualitative and probabilistic proof is layer-owned. The representative scenario set is maintained by META; each row below owns its criterion and threshold.

| Eval ID | Criterion | Threshold |
|---|---|---|
| **DSET-EVAL-PLAN-GOV-001** | Public usability | A new reader can locate current truth, active changes, archive history, templates, schemas, and the governing methodology document without Obsidian or private context |
| **DSET-EVAL-PLAN-GOV-002** | File-only resumability | A fresh evaluator locates active status, next task, proof obligations, PR identity, and archive gates without session context |
| **DSET-EVAL-PLAN-GOV-003** | Migration clarity | A repository with separate existing spec/test/eval/implementation roots reaches one writable `dset/` root without losing test/eval separation or current-truth ownership |
| **DSET-EVAL-PLAN-GOV-004** | Helicopter navigation | A cold reader reaches the owner for methodology, artifact rules, project truth, and agent workflows from the root and area hubs without a filesystem scan |
| **DSET-EVAL-PLAN-GOV-005** | Type and role ownership | Reviewers assign each semantic atom one of four Types and at most one direct subtype, assign documents a separate role/owning question, and avoid duplicate authority or subtype nesting |
| **DSET-EVAL-PLAN-GOV-006** | Specification semantics | Reviewers identify what/why mixing, incomplete entity lifecycles, and definition-before-use violations while treating forward references only as connections |
| **DSET-EVAL-PLAN-GOV-007** | Governance navigation | Every reviewer reaches the registry and single local governing owner and identifies custom/source provenance without treating a wrapper, template, generated installation, index, summary, or cache as normative |
| **DSET-EVAL-PLAN-GOV-008** | Governance diagnostic usefulness | For every invalid-ownership case, reviewers identify the blocking defect and a safe local correction without reading resolver source or inventing a fallback rule |
| **DSET-EVAL-PLAN-GOV-009** | Fail-closed wrapper restraint | Every independent run stops before governed work when ownership cannot resolve, reports the unresolved state, and does not continue from embedded knowledge or remote framework prose |
| **DSET-EVAL-PLAN-GOV-010** | Semantic routing clarity | Every reviewer routes operator authority to Decision; Conflict/Risk/Opportunity to Question; Defect/Gap/Debt to Problem; Test Plan/Evaluation Plan to QA; and treats Test code, Evaluation prompts, tasks, tickets, Changes, releases, and document roles as non-Type structures |
| **DSET-EVAL-PLAN-GOV-011** | ID grammar | Reviewers use the `DSET` project prefix and canonical full direct-subtype kind when present, Type kind when absent, no subtype path, no retired short alias, and never derive identity from directories alone |
| **DSET-EVAL-PLAN-GOV-012** | Source-to-projection clarity | Reviewers distinguish active Decision atoms and direct subtypes, explicit digest-bound evergreen claim fragments, Questions/Problems and evidence, QA definitions/results, and implementation; judge whether each fragment faithfully carries its source consequence; select active authority over a stale or contradictory projection; preserve immutability; and reject implementation-only change that skips compilation |
| **DSET-EVAL-PLAN-GOV-013** | Authority versus assurance | Reviewers distinguish active atomic authority, a current or stale compiled governing projection, valid authority with stale/missing assurance, and evidence or provenance being mistaken for authority; they select the source over a stale projection, block the affected reliance gate, and identify the proportionate recompile or reopen path |
| **DSET-EVAL-PLAN-GOV-014** | Health-view decision usefulness | A cold reviewer identifies the largest actionable coverage/freshness gap, its denominator and exclusions, and the canonical owner without treating the generated view as authority or maximizing artifact counts |
| **DSET-EVAL-PLAN-GOV-015** | Cross-host review portability | Codex, Claude, and a human reviewer can exchange the same bounded packet/report, identify exact reviewed inputs and permissions, and reach explicit finding dispositions without hidden session context or silent fixes |
| **DSET-EVAL-PLAN-GOV-016** | Priority and conflict clarity | Reviewers identify effective priority for every governed artifact; treat Requirement/Contract as sibling Decision subtypes and Conflict as a Question subtype; classify active-atom/projection, absorption, authority/QA, implementation/authority, evidence/evidence, and generated/canonical differences correctly; select claims only where permitted; and avoid evidence-as-authority or age-based precedence |
| **DSET-EVAL-PLAN-GOV-017** | Flat Type clarity | Independent reviewers consistently keep exactly four application-level Types; identify one primary claim; split multi-head statements; use the general-Type fallback on irreducible ambiguity; distinguish all direct subtypes, especially Contract/Constraint/Requirement and Test Plan/Evaluation Plan; separate directive/acceptance/carrier and QA plan/execution/result/evidence; never nest User Story under Requirement, classify Risk as Problem, or hide Defect/Gap as Debt; avoid Conflict for ordinary drift, failed assurance, nonconformance, or wording differences; and correct the earliest ambiguous owner rather than averaging disagreement |
| **DSET-EVAL-PLAN-GOV-018** | Rationale usefulness without ceremony | Reviewers add concise rationale to Decisions and other atoms when it materially helps review, investigation, interpretation, or replacement; omit it when it adds only placeholder prose; and never treat rationale as a hidden Requirement, Contract, lifecycle event, or evidence owner |
| **Historical DSET-EVAL-PLAN-GOV-019** | MECE artifact classification | Delivery-name interpretation is superseded by `DSET-EVAL-PLAN-GOV-029` |
| **DSET-EVAL-PLAN-GOV-020** | Artifact-name readability | A cold reviewer identifies the primary artifact role from the base ID/filename without mistaking an optional subtype for the stable classification axis, and can find the subtype inside the artifact when needed |
| **DSET-EVAL-PLAN-GOV-021** | One-level architecture comprehension | From any project, feature-group, feature, or layer hub, a cold reviewer can identify the immediate structure below it and route to the correct owner without needing a single unreadable whole-system diagram or treating the visualization as authority |
| **DSET-EVAL-PLAN-GOV-022** | Structural-scope ownership | Reviewers place each claim at the narrowest common structural ancestor of every affected owner and subject; keep feature-only high-level claims local; place cross-child Contracts, shared semantics, end-to-end QA, cross-cutting policy, integration, release/readiness, and cross-owner Questions or Problems at the correct common scope; and never duplicate child truth into a parent |
| **DSET-EVAL-PLAN-GOV-025** | Artifact-emission precision and promotion restraint | At medium and high strictness, reviewers distinguish material ambiguity from optional unknown detail, ask only questions needed to make an immutable claim safe, identify the narrowest eligible immediate parent scope, and propose rather than silently perform promotion or broad generalization |
| **DSET-EVAL-PLAN-GOV-026** | Typed-relation clarity | At least 90% of representative structure, analysis, projection, implementation, QA, evidence, resolution, override, replacement, and fallback cases receive the narrowest correct relation, with no authority or lifecycle error and no use of `child_of` as generic causality or `relates_to` to avoid a known type |
| **DSET-EVAL-PLAN-GOV-028** | Settings discoverability | At least 90% of cold-reader questions about active settings, accepted values, effects, defaults, examples, project-manifest facts, governing definitions, legacy compatibility, and external-format boundaries are answered correctly without an authority-source error |
| **DSET-EVAL-PLAN-GOV-029** | Version-name interpretability | Every independent reviewer identifies Version as the release-lifecycle artifact Type without extra naming explanation; assigns Roadmap, Version Scope, Change, Release Plan, Readiness Record, and Release Record as flat direct subtypes with their distinct roles; and does not confuse the Type with product-version strings, source-control versions, deployment state, or semantic Decision Types |
| **DSET-EVAL-PLAN-GOV-032** | Control/runtime/scratch interpretability | Given a repository tree including hidden entries and DSET filenames, independent reviewers recognize `.dset/` as committed control truth, `.dset_runtime/` as ignored resumable state, `/tmp` or the native Windows temporary root as disposable scratch, same-directory swap state as bounded atomic publication, repository-root-relative stored paths, and file-relative Markdown links without confusing these roles or bases |
| **DSET-EVAL-PLAN-GOV-033** | Numbered layer-order interpretability | Cold reviewers infer META → GOV → TOOL → SKILL → OPS from the five directory names, route artifacts correctly, and do not treat numeric prefixes as semantic IDs, authority precedence, feature hierarchy, or dependency order |
| **DSET-EVAL-PLAN-GOV-034** | Feature/layer distinction | Cold reviewers identify features as peers joined by horizontal Contracts, layers as a forward-only refinement chain, adjacent influence as preferred, and downstream references to upstream truth as consumption rather than backward governance |

## Calibration and evidence

Use at least two independent reviewers for a baseline where interpretation matters. Record disagreements by case and criterion, and correct the earliest ambiguous owner rather than averaging incompatible results.
