# Solution landscape — Add artifact governance profiles

## Capability gaps

| Gap | Required evidence |
|---|---|
| Orthogonal enforcement axes | A project can combine code and artifact profiles without either impersonating the other |
| Helicopter navigation | Root and area hubs route readers without exhaustive leaf inventories |
| Artifact ownership | Each type owns one question and avoids copied authority |
| What/why/how/history separation | Normative, rationale, procedure, and evidence surfaces remain distinct and linked |
| Executable structure | Stable configuration errors fail deterministically while semantic judgment stays qualitative |

## Candidates

| Candidate | Evidence | Decision |
|---|---|---|
| Treat documentation as another language profile | Reuses the existing profile concept but conflates toolchain/language gates with knowledge architecture | Reject |
| One large documentation standard | Centralizes rules but recreates the navigation and mixed-ownership problem | Reject |
| Author-owned knowledge-governance patterns | Existing practice separates hubs, active rules, rationale, playbooks, decisions, and history, with parent-tree validation | Adapt |
| DSET-native artifact profiles plus GitHub hubs | Preserves the useful separation while using portable Markdown links and repository-owned schemas/validation | Build |

## Decision

Borrow the concepts of thin hubs, one active control plane, atomic ownership, separate rationale/procedure/history, and structural hierarchy checks. Do not copy the vault's private paths, Obsidian frontmatter, wiki links, backlink dependency, numerical folder taxonomy, or application-specific metadata into the public framework.
